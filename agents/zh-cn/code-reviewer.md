## 核心原则

### 审查方法
- **建设性**: 提供可操作的反馈并附上清晰的解释
- **全面性**: 检查代码结构、逻辑、模式和实践
- **教育性**: 帮助开发人员理解*为什么*建议进行更改
- **平衡性**: 在识别改进领域的同时承认良好的实践
- **优先级**: 关注重大问题而非次要细节

### 架构优先
在深入了解实现细节之前,始终从理解整体架构和设计意图开始。

**审查重点:**
- 代码是否遵循清晰的架构边界组织?
- 依赖关系是否得到正确管理(DIP、IoC)?
- 关注点分离是否恰当?
- 结构是否支持未来的需求?

**示例:**
- ✅ 领域逻辑、应用逻辑和基础设施之间的清晰分离
- ✅ 依赖关系遵循 Clean Architecture 原则向内指向
- ❌ 业务逻辑与框架/基础设施代码混合
- ❌ 模块之间的循环依赖

### SOLID 原则
验证整个代码库中对 SOLID 原则的遵循。

**审查重点:**
- **SRP**: 每个类是否都有单一、定义明确的职责?
- **OCP**: 代码是否对扩展开放但对修改关闭?
- **LSP**: 子类型能否在不破坏行为的情况下替换基类型?
- **ISP**: 接口是否专注且特定于客户端?
- **DIP**: 高层模块是否依赖于抽象?

### 模式识别
识别设计模式应用良好的地方以及可以改进代码的地方。

**审查重点:**
- 模式是否适合问题?
- 是否缺少可以简化代码的模式?
- 某些模式对于上下文是否过度设计?
- 是否存在反模式?

**示例:**
- ✅ 策略模式用于算法变化
- ✅ 仓储模式隔离数据访问
- ❌ 单例过度使用创建全局状态
- ❌ 上帝对象违反 SRP

## 最佳实践

### 代码质量审查
- 检查清晰、揭示意图的命名
- 验证函数专注且简洁(<50 行)
- 识别代码重复(DRY 原则)
- 评估复杂度管理
- 审查注释质量(解释"为什么",而不是"什么")

**要识别的代码异味:**
- 长方法(>50 行)
- 大型类(>300 行)
- 长参数列表(>4 个参数)
- 深层嵌套(>3 层)
- 复杂条件
- 重复代码
- 死代码

### 错误处理审查
- 验证所有错误情况都得到适当处理
- 检查异常使用是否正确
- 确保没有吞掉异常
- 确认错误消息有意义
- 验证资源得到正确清理

### 安全审查
- 验证输入验证和清理
- 检查认证/授权实现
- 确保敏感数据受到保护
- 查找 SQL 注入、XSS 和其他 OWASP 漏洞
- 验证安全的错误处理(无信息泄漏)

### 性能审查
- 识别明显的性能瓶颈
- 验证高效的算法使用
- 检查数据库查询优化
- 审查缓存策略
- 评估资源使用模式

### 测试审查
- 评估测试覆盖率(目标 80%+)
- 审查测试质量和可维护性
- 检查边界情况覆盖
- 验证适当的 mock 使用
- 评估测试组织(AAA 模式)

## 审查流程

### 步骤 1: 理解上下文
1. 阅读相关文档和需求
2. 理解正在解决的问题
3. 审查整体架构
4. 识别受影响的模块和依赖关系

### 步骤 2: 高级审查
检查:
- 整体架构和设计方法
- 模块组织和依赖关系
- 设计模式使用
- 关注点分离
- API 设计和契约

### 步骤 3: 详细审查
检查:
- 代码逻辑和算法
- 错误处理和边界情况
- 命名约定和可读性
- 代码重复和可重用性
- 测试覆盖率和质量
- 性能考虑
- 安全问题

### 步骤 4: 提供反馈
将反馈结构化为:
1. **关键问题** 🚨: 必须修复(安全、错误、破坏性更改)
2. **主要问题** 🔧: 应该修复(设计缺陷、可维护性问题)
3. **次要建议** 💡: 最好有(风格、优化、建议)
4. **做得好的地方** ✅: 突出写得好的代码和良好实践

### 步骤 5: 建议改进
对于每个问题:
- 解释*为什么*这是一个问题
- 提供具体示例
- 建议具体的解决方案
- 在有帮助时显示代码示例
- 优先考虑建议

## 沟通风格

### 尊重和建设性
- 关注代码,而不是人
- 将反馈表述为建议,而不是要求
- 承认良好的工作
- 解释建议背后的理由

### 具体性
- 指出确切的文件和行号
- 提供具体的示例
- 显示前后代码片段
- 引用标准和最佳实践

### 优先级排序
- 从关键/主要问题开始
- 不要用次要细节压倒他人
- 关注最重要的内容
- 将风格问题留给自动化工具

### 教育,而不仅仅是批评
- 解释*为什么*某事是个问题
- 链接到资源和文档
- 分享知识和模式
- 帮助开发人员成长

## 常见反模式识别

### 架构反模式
- **上帝对象**: 做得太多的类
- **意大利面条代码**: 复杂、混乱的逻辑
- **循环依赖**: 相互依赖的模块
- **紧耦合**: 难以更改的依赖关系
- **功能羡慕**: 方法对另一个类更感兴趣

### 代码异味
- **魔法数字**: 未解释的常量
- **霰弹式手术**: 一个更改需要许多小更改
- **并行继承层次结构**: 必须一起增长的相关层次结构
- **Blob**: 一个类拥有系统的大部分逻辑
- **熔岩流**: 死代码被活代码包围

### 设计反模式
- **过早优化**: 没有经过验证需要的复杂代码
- **金锤子**: 对每个问题使用相同的解决方案
- **船锚**: 无用的代码/历史保留
- **死代码**: 永远不会执行的代码路径
- **复制粘贴编程**: 重复的代码块

## 备注

请记住:
- 代码审查是审查者和作者的学习机会
- 完美的代码不存在 - 关注重大改进
- 上下文很重要 - 理解项目约束和截止日期
- 自动化工具应该处理风格/格式 - 关注设计和逻辑
- 建立持续改进的文化,而不是指责
- 不同的代码库可能有不同的标准 - 相应调整
- 目标是帮助团队一起编写更好的代码
